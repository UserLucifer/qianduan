# 后端错误码文档

更新时间：2026-05-03

## 响应结构

后端业务接口统一返回：

```json
{
  "code": 0,
  "message": "成功",
  "data": {},
  "timestamp": "2026-05-03 10:30:00"
}
```

- `code = 0` 表示成功。
- `code != 0` 表示失败，前端应优先按 `code` 做分支，不要依赖 `message` 文案。
- `message` 可直接用于 toast，但前端可按 `code` 覆盖为更适合页面语境的提示。
- HTTP 状态仍然有意义：`401` 处理登录态，`403` 处理权限，`409` 多为重复提交或并发冲突，`422` 多为表单校验。

## 前端通用处理

| 场景 | 建议处理 |
| --- | --- |
| `401` 或 `code=20012/20011/80004` | 清理本地 token，跳转登录或打开登录弹窗 |
| `403` | 提示无权限或账号不可用 |
| `409` | 禁用重复点击，提示刷新后重试或展示“正在处理” |
| `422` | 读取 `data` 字段展示字段级表单错误 |
| `5xx` | 弹出系统繁忙，保留重试入口 |

## 兼容说明

- `10001-10004`、`11001-11004` 为历史已使用错误码，继续保留。
- 本次专项后，业务接口会尽量返回模块化错误码；前端新增逻辑请优先判断细分码。
- 仍可能出现 `BUSINESS_ERROR`、`BAD_REQUEST` 等通用码，通常来自未专项覆盖的兜底场景。
